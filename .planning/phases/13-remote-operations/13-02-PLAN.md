**StatusBar.svelte** -- permanent bottom bar, always visible when a repo is open:

Props: `repoPath: string`

State:
- `progressLine: string` -- latest progress text from remote-progress event
- `isRunning: boolean` -- true while a remote op is active
- `error: { code: string; message: string } | null` -- persists until next op starts

Behavior:
- Listen to `remote-progress` events via `listen()` from `@tauri-apps/api/event`. On each event where `event.payload.path === repoPath`, set `progressLine = event.payload.line` and `isRunning = true`. Clean up listener on destroy.
- Expose a `startOp()` function (called by parent or via binding/callback) that clears any previous error and sets isRunning=true. Alternatively, detect op start when the first remote-progress event arrives.
- Expose a `finishOp(error?: TrunkError)` function that sets isRunning=false, clears progressLine, and optionally sets the error state.
- Better approach: use a shared reactive store or pass callbacks. Simplest: export `startOp`/`finishOp` methods via `$bindable` or have the Toolbar call them directly. Use a module-level writable store `remoteOpState` in a new file `src/lib/remote-state.ts` with `{ isRunning: boolean, error: TrunkError | null, progressLine: string }` as a `$state` rune. Both StatusBar and Toolbar import this store.

Create `src/lib/remote-state.ts`:
```typescript
import type { TrunkError } from './invoke.js';

export const remoteState = $state({
  isRunning: false,
  progressLine: '',
  error: null as TrunkError | null,
});
```

StatusBar reads from `remoteState`. Toolbar writes to `remoteState`.

**NOTE on $state in .ts files:** Svelte 5 runes ($state, $derived) require the file extension `.svelte.ts` to be processed by the Svelte compiler. Use `src/lib/remote-state.svelte.ts` instead of `.ts`.

Layout:
- Fixed height bar (24-28px) at bottom of window
- Background: `var(--color-surface)`, border-top: `1px solid var(--color-border)`
- Three sections: left (status text), center (progress), right (cancel button)
- Idle: show branch name and remote info (e.g., "main -> origin/main") or last operation result -- Claude's discretion
- Running: show spinner (CSS animation, rotating circle or dots) + progressLine text
- Error: show error icon + message text styled with `var(--color-error)` or a warning color
- Cancel button: only visible when `isRunning === true`. Small "X" button. On click: `safeInvoke('cancel_remote_op')`, then set isRunning=false.

Error message mapping (per user decisions):
- `auth_failure` -> "Authentication failed -- check your SSH key or credential helper"
- `non_fast_forward` -> "Push rejected (non-fast-forward)" with a clickable "Pull now" span/button. On click: trigger `git_pull` via safeInvoke (calls the pull handler from Toolbar or directly).
- `no_upstream` -> Pass through the raw git error message (per user decision: "pass through git's native error")
- `remote_error` -> Show the raw error message

**Toolbar.svelte** -- GitKraken-style centered toolbar:

Props: `repoPath: string`

Buttons (centered horizontally):
- **Pull** (with chevron): Default click triggers `safeInvoke('git_pull', { path: repoPath })` (no strategy, respects gitconfig). Chevron on right side opens PullDropdown.
- **Push**: triggers `safeInvoke('git_push', { path: repoPath })`
- **Branch**: opens InputDialog for create-branch (reuse Phase 12 InputDialog pattern -- set a dialogConfig state, render InputDialog conditionally). On confirm: `safeInvoke('create_branch', { path: repoPath, name: inputValue, checkout: true })`.
- **Stash**: triggers `safeInvoke('stash_save', { path: repoPath })`
- **Pop**: triggers `safeInvoke('stash_pop', { path: repoPath })`

All remote buttons (Pull, Push, Fetch via dropdown) are DISABLED when `remoteState.isRunning === true`. Branch, Stash, Pop remain enabled (they are local ops).

Each remote button handler:
1. Set `remoteState.isRunning = true; remoteState.error = null; remoteState.progressLine = '';`
2. Call safeInvoke for the command
3. On success: set `remoteState.isRunning = false; remoteState.progressLine = '';`
4. On error (catch): set `remoteState.isRunning = false; remoteState.error = caught error;`
5. After success of any remote op: the backend already emits repo-changed, so no explicit graph refresh needed from frontend.

After Stash/Pop/Branch complete successfully, emit needs refresh. These commands already emit `repo-changed` from the backend, so no additional frontend work needed.

Styling:
- Height: ~36px, background: `var(--color-surface)`, border-bottom: `1px solid var(--color-border)`
- Buttons centered using flex with justify-center
- Each button: small, compact, text label (no icons required -- Claude's discretion on whether to add simple Unicode icons like arrows)
- Disabled state: opacity 0.5, pointer-events none

- "Fast-forward only" -> `safeInvoke('git_pull', { path: repoPath, strategy: 'ff-only' })`
- "Pull (rebase)" -> `safeInvoke('git_pull', { path: repoPath, strategy: 'rebase' })`

Implementation:

- Each option is a button. On click: close dropdown, set remoteState.isRunning, call safeInvoke, handle success/error same as Toolbar buttons.
- Disabled when `remoteState.isRunning === true`
